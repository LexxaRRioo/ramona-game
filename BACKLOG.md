## Bugs (known, not yet fixed)

- [x] Lie-down/curl-into-sleep transition should pick one side (left-curl or right-curl) at random each time and stick to frames from only that side - row 6's original 10-frame sequence mixed both curl directions. Split into `lieDownLeft`/`lieDownRight` (+ matching `sleepLeft`/`sleepRight`), one picked at random per settle - pending release.
- [x] Ramona sits in front of the Dock instead of on it - the Dock's AXList frame from Accessibility reports ~5pt more height than the strip's actual rendered top edge (confirmed via screenshot pixel-sampling), so she settled floating just above/in front of the visible glass. Fixed with a measured correction in `Dock.bottomFrame()` - pending release.

- [ ] Window/Dock floor tracking is fragile and keeps regressing - test-covered and fixed this session, but NOT yet verified live (leave unchecked until confirmed - see last bullet below):
  - Root cause 1 (window tracking): `FrontmostWindowTracker` only ever tracked "whichever window is currently frontmost", with no concept of window identity - so if she'd climbed onto window A and the user then focused window B (without moving A), the tracker silently started reporting B's frame through the exact same callback used for "your window moved", and she'd follow it. Fixed by having the tracker compare AXUIElements across reports (`CFEqual`) and pass an `isSameWindow` flag through `onFrameChange`; `FloorTracking.nextSurface` (new, pure, unit-tested in `FloorTrackingTests`) decides the resulting surface, and `CatScene` uses `isSameWindow` to pick the animation - a same-window move re-settles normally, a different window becoming frontmost plays an explicit walk-style jump (`playJumpToCurrentSurface`) instead of silently gliding across in whatever pose she was in.
  - Root cause 2 (Dock sinking/floating): `Dock.topPaddingCorrection` was a fixed 5pt guessed from one machine's Dock. The padding above the Dock's visible icons scales with icon size (System Settings > Dock > Size), so a constant can't generalize - this dev machine's Dock (tile size 66, magnification on) is a different config than whatever 0.2.0 was tuned against. Replaced with a direct measurement: `Dock.bottomFrame()` now unions the individual icon elements' own AX frames (median of their top edges, robust to a single magnified/hovered outlier) instead of applying a constant to the outer list frame - see `Dock.strip`/`Dock.median`, unit-tested in `DockGeometryTests`.
  - Extracted `Surface`/`FloorTracking` (window > Dock > floor ground-line priority, perch validity, and the surface-transition decision) out of `CatScene` into a plain, AppKit/SpriteKit-free type specifically so this logic has real test coverage instead of only being checkable by eyeballing the running app.
  - **Not independently verified live**: pixel-level Dock alignment needs Accessibility permission granted to a running build, which isn't something achievable from this sandboxed session - build via `Scripts/build-dev-app.sh`, grant Accessibility once to `.build/Ramona.app`, and confirm she now sits cleanly on the Dock strip (not floating or sinking) and correctly jumps between windows as focus changes.
- [x] Debug menu "Force Action": confirmed via code reading (not just inspection - traced the full call chain) that it already applies immediately - `Picker.onChange` -> `AppDelegate.forceAction` -> `BehaviorEngine.setForcedAction` -> `evaluateAction()` -> `onStateChange` -> `CatScene.apply` all run synchronously on the main thread with no Timer/dispatch in between. No fix needed - pending release.
- [x] Dev-only toggle to hide the sprite/overlay entirely while Claude is working in the repo - `RAMONA_HIDDEN=1` env var now skips creating the overlay window in `AppDelegate.applicationDidFinishLaunching`, while the behavior engine/accessibility flow still start normally - pending release.

Added `BehaviorEngine.resumeAfterHold()`, which always resyncs the view. Pending release.
- [ ] Climb has never actually been watched/verified live - it's trait/mood-scored like any other action (CatAction.climb.score, gated on `windowAvailable`) and reuses the walk animation with no dedicated climb clip (see CatScene's `case .walk, .climb:` comment - "a distinct climbing animation is a later art thing"), so it's unclear it looks/feels right in practice. Wanted behavior: rather than climb being just one more scored candidate that may or may not win, she should actively evaluate whether there's a reachable "nearest highest point" and prefer it - e.g. right after launch, starting on the Dock, she should look at the current active window (FrontmostWindowTracker already tracks exactly that single window) and head up onto it if it's higher and reachable, instead of only climbing when the utility AI happens to pick .climb this tick. Needs test coverage for the targeting/preference logic, not just a live look at the animation.
- [ ] When sitting/standing on a window (or the menu bar strip) would leave her partially hidden behind other UI on top of it (screenshot this session: sitting on the menu bar, a status item/notch icon overlapping and clipping her), she should lie down instead - a lying pose takes less vertical height, so it's more likely to fit fully visible in whatever gap she's perched in. Needs a way to detect "would I be occluded here" (or at least "is there enough clear vertical room in this gap") and a lying-idle pose to switch to - ties into the unused lying-idle frames noted in the Animation section above.
- [x] Rework mouse interaction: left click (and click-hold-move) should be petting/cursor-petting, not pickup. Right click should move/drag her instead. Swapped `CatScene`'s handlers - `mouseDown`/`mouseDragged`/`mouseUp` (left) now pet on click-release and repeatedly (throttled to petPulse's 0.3s cadence) while held-and-moved; `rightMouseDown`/`rightMouseDragged`/`rightMouseUp` carry the pickup-drag mechanics that used to live on the left button, unchanged otherwise - pending release.
- [x] Disable leg movement during playJumpToCurrentSurface's jump (and any future jump animations) - hold a static leap pose instead of cycling the run clip's legs mid-air. Added `CatSprites.leapRight`/`leapLeft`, the run cycle's own col-1 mid-leap frame held static, swapped in for the jump only (seekAttention's run loop is untouched) - pending release.
- [ ] Behavior tuning pass: 3x less walking, 2x more sleeping, 1x more grooming - relative frequency, not absolute. Currently `CatAction.score`'s per-action formulas (`.walk`: `(1 - needs.play) * (0.5 + traits.playfulness) + traits.boldness * 0.2`; `.sleep`: `(1 - needs.energy) * (0.5 + traits.laziness)` (+0.4 in-window); `.groom`: `0.3 + needs.play * 0.25 + needs.energy * 0.1`) are the tunable weights - shifting relative frequency means adjusting these against each other (and possibly `NeedsState.decay`'s play-fill-while-active rate, which is what makes walk's score drop off), not just multiplying one in isolation, since they all compete against the same `actionSwitchMargin` each tick.
